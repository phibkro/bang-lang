# Bang v0.5 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Align the compiler with the thunk axiom (everything is a description, `!` forces), add effect handling (`.handle`/`.catch`/`.map`/`.tap`), resource management (`use`), push subscriptions (`on`), nested patterns + guards, newtype/record types, and compile-time evaluation (`comptime`).

**Architecture:** Seven waves, ordered by dependency. Each wave follows the pipeline: AST → Lexer → Parser → Interpreter (ground truth) → Tests → Checker → Codegen → Formatter. The interpreter defines semantics first. The compiler translates those semantics. Monorepo: `@bang/core` (interpreter domain — AST, Parser, Interpreter, Formatter), `@bang/compiler` (Checker, Codegen, pipeline composition).

**Tech Stack:** TypeScript, Effect (Schema.TaggedClass, Match.exhaustive, HashMap, Data.taggedEnum), @effect/vitest, @effect/printer

**Spec:** `docs/superpowers/specs/2026-03-28-bang-v05-design.md`

```typescript
Match.tag("BinaryExpr", (e) =>
  Effect.gen(function* () {
    // Special case: <- does NOT evaluate left side normally
    if (e.op === "<-") {
      if (e.left._tag !== "Ident") {
        return yield* Effect.fail(new EvalError({ message: "Left side of <- must be an identifier", span: e.span }));
      }
      const cell = HashMap.get(env, e.left.name);
      if (Option.isNone(cell) || cell.value._tag !== "MutCell") {
        return yield* Effect.fail(new EvalError({ message: `Cannot mutate non-mutable binding: ${e.left.name}`, span: e.span }));
      }
      const newValue = yield* evalExpr(e.right, env);
      cell.value.ref.value = newValue;
      return newValue;
    }
    // Normal binary: evaluate both sides
    const left = yield* evalExpr(e.left, env);
    const right = yield* evalExpr(e.right, env);
    return yield* applyBinaryOp(e.op, left, right, e.span);
  }),
),
```

## Wave 2: Dot Methods + Effect Handling

### Task 2: Interpreter — Effect Handler Stack

**Files:**

- Modify: `packages/core/src/Interpreter.ts`
- Modify: `packages/core/src/Value.ts`
- Create: `packages/compiler/test/EffectHandling.test.ts`

This is the most complex task. The interpreter needs a real effect system.

- [ ] **Step 1: Extend the interpreter environment**

Add a handler context alongside the env. The interpreter currently passes `Env` (HashMap of bindings). Extend to pass a `Context` that includes both env and handler stack:

```typescript
interface InterpreterContext {
  readonly env: Env;
  readonly handlers: HashMap.HashMap<string, Value>; // type name → implementation value
  readonly catchers: ReadonlyArray<{ tag: string; handler: Value }>; // error catchers
}
```

Or simpler: keep `Env` as-is and add handlers as a second parameter threaded through `evalExpr`. Choose the approach that requires fewer signature changes — likely extending Env to include handler bindings (handlers are just named values in scope with a special prefix like `__handler_Console`).

**Recommended approach:** Keep it simple. `.handle Console impl` evaluates to a value that wraps the inner expression with handler context. The simplest model:

1. When `DotAccess(expr, "handle")` is applied with args `[TypeName, impl]`:
   - Evaluate `expr` with `impl` bound to `TypeName` in env
   - When the inner expression calls `!TypeName.operation(args)`, look up `TypeName` in env, get the impl record, call the operation

2. This works because effect interfaces are just types (records). If `Console` is in env as a record value, then `Console.log` is just field access + application. The `.handle` binds the implementation.

- [ ] **Step 3: Implement .map and .tap in interpreter**

In `evalExpr`, the `DotAccess` handler currently fails with "DotAccess not supported." Replace it:

When `DotAccess(object, field)` where `field` is `"map"` or `"tap"`:

- Evaluate the object
- Return a special value (e.g., `Closure` that captures the object result and applies the argument function)

Or handle it in the `App` handler: when `App(DotAccess(expr, "map"), [f])`:

- Evaluate expr to get value
- Evaluate f
- Apply f to value
- Return result

For `.tap`: same but return original value instead of f's result.

- [ ] **Step 4: Implement .handle in interpreter**

When `App(DotAccess(expr, "handle"), [typeName, impl])`:

- `typeName` resolves to a type name (e.g., "Console")
- `impl` is the implementation value (a record/Tagged value)
- Evaluate `expr` in an env where `typeName` maps to `impl`
- This means DotAccess operations like `!Console.log msg` will find `Console` in env, access `.log`, apply `msg`

This is the simplest possible effect handling — it's just binding a name in the evaluation context. The interpreter doesn't need a formal handler stack because the env already IS a stack (blocks create new scopes).

- [ ] **Step 5: Implement .catch in interpreter**

When `App(DotAccess(expr, "catch"), [errorTag, handler])`:

- Evaluate `expr`
- If it fails with an `EvalError` whose message or tag matches `errorTag`, apply `handler` to recover
- If it succeeds, return the value

Use `Effect.catchAll` or `Effect.either` to intercept failures.

- [ ] **Step 5b: Implement .match as dot method**

The spec says `match expr { arms }` desugars to `expr.match { arms }`. Both forms must work.

In the interpreter, when `App(DotAccess(expr, "match"), [handlerBlock])`:

- Evaluate `expr` to get the scrutinee value
- The handler block contains match arms — evaluate the match logic as in `MatchExpr`

In the parser, `.match { arms }` already parses as `App(DotAccess(expr, "match"), [Block(...)])`. The interpreter and codegen need to detect this pattern and route to match evaluation.

In codegen, `expr.match { arms }` generates the same `Match.value(expr).pipe(...)` as `match expr { arms }`.

Run: `npx vitest run packages/compiler/test/EffectHandling.test.ts`

- [ ] **Step 7: Implement dot methods in Codegen**

In `packages/compiler/src/Codegen.ts`, update `emitExpr` for `App(DotAccess(expr, method), args)`:

```typescript
// Detect dot-method patterns
if (e.func._tag === "DotAccess") {
  const method = e.func.field;
  const obj = emitExpr(e.func.object, decls, mutNames, hoisted);
  if (method === "handle" && e.args.length === 2) {
    const typeName = emitExpr(e.args[0], decls, mutNames, hoisted);
    const impl = emitExpr(e.args[1], decls, mutNames, hoisted);
    return `pipe(${obj}, Effect.provide(Layer.succeed(${typeName}, ${impl})))`;
  }
  if (method === "catch" && e.args.length >= 1) {
    const tag = emitExpr(e.args[0], decls, mutNames, hoisted);
    const handler =
      e.args.length > 1 ? emitExpr(e.args[1], decls, mutNames, hoisted) : "(_) => Effect.void";
    return `pipe(${obj}, Effect.catchTag(${tag}, ${handler}))`;
  }
  if (method === "map" && e.args.length === 1) {
    return `pipe(${obj}, Effect.map(${emitExpr(e.args[0], decls, mutNames, hoisted)}))`;
  }
  if (method === "tap" && e.args.length === 1) {
    return `pipe(${obj}, Effect.tap(${emitExpr(e.args[0], decls, mutNames, hoisted)}))`;
  }
}
```

Add `Layer` and `pipe` to imports when dot methods are present.

- [ ] **Step 3: Implement `use` in interpreter**

`use x = f` in the interpreter: evaluate `f` to get a resource provider (a Closure that takes a callback). The resource is acquired when forced. In the interpreter's eager model, `!use x = f; rest` desugars to calling `f` with a callback containing `rest`.

Handle `UseExpr` in `evalExpr`:

```typescript
Match.tag("UseExpr", (e) =>
  Effect.succeed(/* return a thunk value representing the use binding */),
),
```

When `Force(UseExpr(...))` is encountered in block evaluation, the interpreter needs to:

1. Evaluate `f` (the resource provider)
2. Call `f` with a callback that evaluates the rest of the block
3. The callback receives the resource as `x`

This requires the block evaluator to detect `ForceStatement(Force(UseExpr(...)))` and restructure remaining statements into the callback. This is the most complex part of `use`.

**Interpreter approach:** The interpreter must implement the callback semantics to remain ground truth. When the block evaluator encounters `Force(UseExpr(name, f))` as a statement:

1. Evaluate `f` to get the resource provider (a Closure taking a callback)
2. Gather all remaining statements + the block's return expression into a synthetic lambda body
3. Call `f(callback)` where callback binds `name` to the resource
4. This matches the desugaring: `!use x = f; rest` → `!f (x) -> { rest }`

The block evaluator already processes statements sequentially. When it hits a `!use`, it must consume the rest of the block as the callback. This is similar to how CPS transforms work — the interpreter restructures at eval time.

If this proves too complex for v0.5, an acceptable compromise: `!use x = f` evaluates `f()` (calling the provider with no cleanup), binds result to `x`, and the property test for `use` is excluded from `eval === run(codegen)` verification. Document as a known interpreter limitation.

**Codegen:** Restructure block: everything after `!use x = f;` becomes the callback body.

- [ ] **Step 1: Parser disambiguation for newtype**

In `parseTypeDecl`, after parsing `type Name =`, the current code expects a constructor (TypeIdent). Newtypes also start with TypeIdent but are followed by a non-constructor token (not `|`, not another TypeIdent, not `{`).

Disambiguation logic after `=`:

1. If TypeIdent followed by `|` → ADT (already handled)
2. If TypeIdent followed by TypeIdent or Ident (type params of constructor) → ADT (already handled)
3. If TypeIdent alone, followed by EOF/newline/semicolon/next-statement-start → **newtype**
4. If `{` with `Ident :` → record type (Task 8)

For newtype: store as `TypeDecl` with a single `PositionalConstructor` named after the type, arity 1 (wraps one value of the target type). Or add a new AST variant for newtypes.
