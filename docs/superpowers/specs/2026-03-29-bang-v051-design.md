# Bang v0.5.1 Design Spec — Deepen + Polish

## Scope

8 items: 6 deepening existing features, 2 language UX improvements.

### A) Deepen v0.5

#### 1. Record field access (`user.name`)

**Value.ts:** Add a `Record` variant to the Value TaggedEnum that stores named fields:

```typescript
Record: {
  readonly tag: string;
  readonly fields: HashMap.HashMap<string, Value>;
};
```

Or simpler: extend `Tagged` to optionally carry field names. Simplest approach: add `fieldNames: ReadonlyArray<string>` to `Tagged`:

```typescript
Tagged: {
  readonly tag: string;
  readonly fields: ReadonlyArray<Value>;
  readonly fieldNames: ReadonlyArray<string>;  // empty for positional, populated for records
};
```

When `RecordTypeDecl` registers a constructor, the resulting `Tagged` values carry field names. When DotAccess evaluates `obj.fieldName` on a Tagged value with `fieldNames`, find the index of `fieldName` in `fieldNames`, return `obj.fields[index]`.

This keeps field metadata in the value itself — no env lookup needed. A Tagged value that escapes its defining scope (returned from a function, stored in a data structure) still knows its field names.

Update all existing `Tagged` construction sites to include `fieldNames: []` (empty for ADT positional fields, populated for records).

**Interpreter:** When evaluating a record constructor application, produce `Tagged({ tag, fields: [...values], fieldNames: [...names] })`.

**Tests:** `type User = { name: String, age: Int }\nu = User "alice" 30\nx = u.name` → `Str("alice")`.

#### 2. `on` returns Subscription with `.abort`

**Interpreter:** `OnExpr` handler returns `Tagged({ tag: "Subscription", fields: [abortClosure], fieldNames: ["abort"] })`. The abort closure is constructed by capturing both the `subscribers` array reference (from `MutCell.ref.subscribers`) and the specific subscriber function. When the abort closure is forced, it uses `Array.prototype.indexOf` + `Array.prototype.splice` to remove the subscriber from the array.

```typescript
const sub = (newValue: Value) => applyValue(handler, [newValue], span);
sourceCell.ref.subscribers.push(sub);
// Build abort closure that removes this specific subscriber
const abortFn = Closure({
  params: [],
  body: /* synthetic — just remove sub from array */,
  env: /* capture sub reference */,
});
```

Since we can't easily build synthetic AST for the abort body, use a simpler approach: add a `BuiltinFn` variant to Value, or store a boolean flag `{ active: true }` that the subscriber checks before firing, and the abort sets to `false`.

**Recommended:** Wrap the subscriber in `{ active: true, fn: sub }`. Abort sets `active = false`. The mutation handler checks `active` before calling. No new Value variants needed.

**DotAccess:** When accessing `.abort` on a Tagged value with tag `"Subscription"`, return the first field (the abort closure/thunk).

**Tests:** Subscribe, mutate (fires handler), abort, mutate again (handler does NOT fire).

#### 3. `use` resource protocol (CPS in interpreter)

**Important:** The CPS transform happens in the `Block` match arm of `evalExpr` (lines 139-146 of Interpreter.ts), NOT in `evalStmt`. Only the Block handler has visibility into remaining statements + the return expression.

**Current Block handler:** Sequential `for` loop over statements, then eval the return expression.

**New Block handler:** Replace the sequential loop with an index-based approach. At each index, check if the current statement is `ForceStatement(Force(UseExpr(name, f)))`. If so:

1. Evaluate `f` to get the resource provider
2. Build a synthetic continuation: `Block(statements[index+1..], returnExpr)` — a new Block AST node containing the remaining statements and the original return expression
3. Build a lambda: the continuation block becomes the body, `name` is the parameter
4. Apply the provider to this lambda: `applyValue(provider, [lambda])`
5. Return the provider's result (which executes the continuation with the resource bound)

```typescript
Match.tag("Block", (e) =>
  Effect.gen(function* () {
    let blockEnv = env;
    for (let i = 0; i < e.statements.length; i++) {
      const stmt = e.statements[i];
      // Detect !use pattern
      if (stmt._tag === "ForceStatement" && stmt.expr._tag === "Force" &&
          stmt.expr.expr._tag === "UseExpr") {
        const useExpr = stmt.expr.expr;
        const provider = yield* evalExpr(useExpr.value, blockEnv);
        // Build continuation from remaining statements + return expr
        const remaining = e.statements.slice(i + 1);
        const contBlock = new Ast.Block({ statements: remaining, expr: e.expr, span: e.span });
        const contLambda = new Ast.Lambda({ params: [useExpr.name], body: contBlock, span: e.span });
        const callback = yield* evalExpr(contLambda, blockEnv);
        return yield* applyValue(provider, [callback], e.span);
      }
      blockEnv = yield* evalStmt(stmt, blockEnv);
    }
    return yield* evalExpr(e.expr, blockEnv);
  }),
),
```

If `f` is not a Closure (plain value), the `applyValue` call will fail. Fall back: detect non-Closure providers and bind directly (current behavior).

**Tests:**
```bang
withResource = (callback) -> { !callback "acquired" }
result = { !use x = withResource; x }
-- result should be "acquired"
```

#### 4. Nested pattern codegen

**Key constraint:** Effect's `Match.tag("Ok", ...)` consumes ALL `Ok` cases in one callback. You can't call `Match.tag("Ok", ...)` twice. Multiple arms matching the same outer constructor must be **grouped** into a single `Match.tag` callback containing a nested match.

**Algorithm:**
1. Group match arms by their outer constructor tag
2. For each group, if there's only one arm with no nested constructor patterns → emit flat `Match.tag`
3. If a group has multiple arms (or nested constructor patterns) → emit `Match.tag` with an inner `Match.value` on the sub-fields

```typescript
// match result { Ok (Some v) -> v, Ok None -> 0, Err e -> 0 }
// Group: Ok → [Ok (Some v) -> v, Ok None -> 0], Err → [Err e -> 0]

Match.value(result).pipe(
  Match.tag("Ok", ({ _0 }) =>
    Match.value(_0).pipe(
      Match.tag("Some", ({ _0: v }) => v),
      Match.tag("None", () => 0),
      Match.exhaustive
    )
  ),
  Match.tag("Err", ({ _0: e }) => 0),
  Match.exhaustive
)
```

**Implementation:** In codegen, before emitting match arms:
1. Group arms by outer pattern tag (for `ConstructorPattern`)
2. For each group with >1 arm or nested patterns, emit grouped `Match.tag` with inner match
3. Wildcard/binding/literal arms go at the end as `Match.orElse`

**Tests:** Compile `!match (Ok (Some 42)) { Ok (Some v) -> v, Ok None -> 0, Err e -> 0 }` and verify output contains grouped `Match.tag("Ok", ...)` with nested `Match.value`.

#### 5. Flaky roundtrip property test

**Root cause:** The `@effect/printer` `Doc.nest` inside `Doc.group` has a flatten bug. When a lambda appears as a binary operand, the formatter output changes between first and second formatting passes.

**Fix:** The `genBinaryOperand` generator already excludes Lambda and OnExpr. Verify completeness:
1. Run roundtrip test 10 times in a loop. Target: 0 failures across all runs.
2. If still flaky, add further exclusions to generators (e.g., exclude ComptimeExpr from binary operands).
3. Keep `numRuns: 100` per test invocation (current value).
4. Success criterion: 10 consecutive passing runs of the full test suite.

#### 6. `.handle` handler stack

**Current:** `.handle Console impl` binds `Console` in the env and re-evaluates. This works for single handlers but nested handlers on the same type name shadow instead of compose, and there's no scope-based cleanup.

**New:** Thread a handler context through `evalExpr`:

```typescript
interface HandlerContext {
  readonly handlers: HashMap<string, Value>;  // typeName → impl
}
```

When `App(DotAccess(expr, "handle"), [typeName, impl])`:
1. Create a new HandlerContext with `typeName → impl` added
2. Evaluate `expr` with the new context
3. When `!TypeName.operation(args)` is evaluated, look up `TypeName` in the handler context first, then fall back to env

The handler context is a second parameter to `evalExpr` (alongside `env`). It flows through all recursive calls. This is a refactor — `evalExpr` signature changes from `(expr, env)` to `(expr, env, handlers)`.

**Alternatively:** Keep using env but with a namespaced prefix (`__handler_Console`) to avoid collision with user-defined bindings. Simpler, no signature change.

**Recommended:** Namespaced prefix approach for v0.5.1. Less invasive, same behavior for practical purposes. A formal handler stack can come with the type system.

### B) Language UX

#### 7. Braced multi-handler

`.handle { Console -> consoleImpl, Database -> pgDb }` desugars to chained `.handle` calls.

**Parser:** In the `App` handler, when parsing arguments for a DotAccess with field `"handle"` or `"catch"`, if next token is `{`, parse as handler arms: `TypeIdent -> Expr`, separated by `,`. Desugar into chained DotAccess + App nodes:

```
expr.handle { A -> x, B -> y }
→ App(DotAccess(App(DotAccess(expr, "handle"), [A, x]), "handle"), [B, y])
```

This is purely a parser desugar — no new AST nodes. The interpreter and codegen see the chained form.

**Formatter:** Detect chained `.handle` calls and format as braced form. Or: don't — keep the chained form as canonical. Braces are input sugar only.

**Tests:** Parse `expr.handle { Console -> impl, Database -> pgDb }` and verify it produces chained App(DotAccess(...)) structure.

#### 8. `.match` as dot method

`expr.match { arms }` — pipe form of match.

**AST representation:** The parser desugars `.match { arms }` into a synthetic `MatchExpr` at parse time. When the parser sees `DotAccess(expr, "match")` followed by `{` containing match arms, it constructs `MatchExpr({ scrutinee: expr, arms: [...] })` directly. No `App(DotAccess)` pattern to detect downstream — the interpreter and codegen see a normal `MatchExpr`.

**Parser:** In `parseDotAccess` or `parseApplication`, after parsing `expr.match`, check if the next token is `{`. If so, parse the `{` content as match arms (reuse the arm-parsing logic from `parseMatch`), and construct `MatchExpr` with `expr` as scrutinee. If not `{`, fall through to normal application.

This means `.match { arms }` and `match expr { arms }` produce the identical AST. No interpreter or codegen changes needed.

**Tests:** `x = !(Some 42).match { Some v -> v, None -> 0 }` → `Num(42)`. Verify AST is `MatchExpr`.

## Implementation Order

1. Record field access (small, independent)
2. `on` abort (small, independent)
3. Flaky roundtrip fix (small, independent)
4. Handler stack / namespaced prefix (medium, prerequisite for robust .handle)
5. `use` resource protocol (medium, independent)
6. Nested pattern codegen (medium, independent)
7. Braced multi-handler (medium, depends on #4)
8. `.match` as dot method (medium, independent)

Items 1-3 can be done in parallel. Item 4 before 7. Others are independent.

## Deferrals

- Named function sugar — dropped, `f = x -> { ... }` is the canonical form
- `gen { ... }` escape hatch — deferred
- Full handler stack (separate parameter to evalExpr) — deferred to type system work
